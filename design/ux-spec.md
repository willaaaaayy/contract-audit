# Contract Audit — UX-спека: генеративный адаптивный интерфейс

Дизайн строится вокруг одной идеи: **интерфейс не фиксирован, а собирается под конкретный документ и пользователя**. Один и тот же договор юрист, закупщик и согласующий видят по-разному; договор поставки и NDA раскладываются разными блоками; договор с критическим риском «кричит» проблемой сверху, а чистый — сворачивается в короткую сводку. Бэкенд уже отдаёт нужные данные (риски с `severity`/`clauseRef`/`quote`, compliance-findings, ключевые даты и суммы) — задача UI правильно их приоритизировать и разложить.

## 1. Принцип: layout engine, а не статичные экраны

Экран аудита — это не вёрстка, а **список блоков, который вычисляется**. Каждый блок получает вес, блоки сортируются по весу, верхние раскрываются полностью, нижние сворачиваются в правый «rail». Это и есть «генеративный layout»: порядок, плотность и состав блоков — результат функции от контекста, а не константа.

```
weight(block) = base(role) × relevance(contractType) × urgency(riskProfile, deadlines)
```

Входы движка:

- **Тип договора** (классификатор: поставка / аренда / NDA / SLA / услуги / иное) — определяет профильный чек-лист и какие сущности важны (для NDA — срок конфиденциальности и территория; для поставки — сроки оплаты, штрафы, пролонгация).
- **Профиль риска** — суммарный скоринг 0–100 и распределение severity. Чем выше риск, тем плотнее и выше выводятся проблемы; чистый договор сворачивает детали в «всё в порядке».
- **Роль** — юрист / закупщик / согласующий / только-чтение. Меняет видимость блоков и действие по умолчанию (юристу — «предложить правку», согласующему — «согласовать / на доработку»).
- **Состояние документа** — `PENDING` / `PROCESSING` / `DONE` / `FAILED` — управляет прогрессивным раскрытием (скелетоны, частичные результаты, ошибка).

### Инвентарь блоков

| Блок | Источник данных | Когда вес высокий |
|---|---|---|
| `ScoreGauge` — общий скоринг 0–100 | новый агрегат по рискам | всегда виден, размер растёт с риском |
| `RiskList` — найденные риски | `DocumentRisk` | критические/высокие риски → топ |
| `ComplianceFindings` — сверка с политиками | `ComplianceFinding` | есть `MISSING_REQUIRED` по mandatory |
| `KeyDataPanel` — стороны, даты, суммы, юрисдикция | `KeyData` + новые сущности | договоры с близкими дедлайнами |
| `DeadlineTimeline` — таймлайн ключевых дат | `KeyData.dates` | есть дата в пределах окна напоминания |
| `NegotiationSuggestions` — контр-формулировки | новый LLM-вывод | роль = юрист, есть HIGH/CRITICAL |
| `TemplateDiff` — отклонения от эталона | новый | загружен «золотой» шаблон |
| `DocumentViewer` — PDF с подсветкой пункта | blob + `clauseRef`/`quote` | всегда (правая половина на десктопе) |
| `ActionBar` — действия | роль | всегда, закреплён снизу |

Порог раскрытия: блоки с весом выше медианы раскрыты в основной колонке, остальные — свёрнутые карточки в rail, разворачиваются по клику. Это даёт ощущение, что система «сама решила», что показать первым, но пользователь всегда может достать остальное.

## 2. Дизайн-токены

Спокойная enterprise-палитра: нейтральный фон, насыщенный акцент, строгая шкала severity (она же используется в скоринге и в подсветке пунктов — единый язык цвета на всех экранах).

```
--bg:            #0f1115 (dark) / #f7f8fa (light)
--surface:       #171a21 / #ffffff
--surface-2:     #1e222b / #eef1f5
--border:        #2a2f3a / #e2e6ec
--text:          #e6e9ef / #1a1d23
--text-muted:    #9aa3b2 / #5b6573
--brand:         #4f6bed  (индиго — действия, ссылки)
--brand-accent:  #00b3a4  (бирюза — позитив/готово)

severity:
--sev-critical:  #e5484d
--sev-high:      #f08a24
--sev-medium:    #e0b400
--sev-low:       #6e7787
--sev-ok:        #2bb673

radius: 6 / 10 / 14    spacing base: 4px grid
font: Inter / system-ui; mono для clauseRef и сумм
```

Скоринг красится по той же шкале: 0–39 critical-red, 40–59 high-orange, 60–79 medium-amber, 80–100 ok-green. Юрист за 5 секунд считывает «брать или эскалировать».

## 3. Экран аудита договора (ядро)

Десктоп — две зоны: слева **сборка блоков** (генеративная колонка + rail), справа **DocumentViewer** с PDF. Клик по риску подсвечивает соответствующий пункт в PDF (bounding box по `clauseRef`/`quote`) — двусторонняя связь «вывод ↔ первоисточник», главный драйвер доверия.

Поведение по контексту:

- **Высокий риск, роль юрист:** сверху `ScoreGauge` (крупно, красный) → `RiskList` с раскрытым критическим риском и кнопкой «Предложить формулировку» → `NegotiationSuggestions`. KeyData и compliance — в rail.
- **Низкий риск, роль согласующий:** `ScoreGauge` (зелёный, компактный) → одна строка «3 незначительных замечания» → крупная `ActionBar` «Согласовать». Детали свёрнуты.
- **NDA:** `KeyDataPanel` поднимается (срок конфиденциальности, территория, неустойка), профильный чек-лист NDA вместо чек-листа поставки.
- **`PROCESSING`:** скелетоны блоков + потоковое появление рисков по мере готовности; `ScoreGauge` в состоянии «считаю».

Состояния каждого блока: loading (skeleton) / empty («рисков не найдено») / partial / error / collapsed. Риск-карточка: severity-полоса слева, тип, `clauseRef` (mono-чип), цитата (выделена), объяснение, действие по роли.

## 4. PDF-отчёт / экспорт

Отчёт — «застывший» снимок того же визуального языка, пригодный для отправки контрагенту или в дело. Структура: титул (контрагент, тип, дата, скоринг крупно) → сводка по severity → таблица рисков (тип, пункт, цитата, объяснение, рекомендация) → compliance-раздел (противоречия + отсутствующие обязательные политики) → ключевые даты и суммы → приложение со страницами PDF и подсветкой. Форматы: PDF (отчёт), DOCX с track changes (правки в текст договора), XLSX (реестр рисков для интеграций). Перед экспортом — конфигуратор: что включить, обезличить ли, язык.

## 5. Дашборд портфеля

Адаптируется под роль: руководителю — агрегаты (активных договоров, сумма обязательств, концентрация рисков, тепловая карта severity по контрагентам), юристу — «моя очередь» (что назначено, что эскалировано). Карточки-метрики сверху, ниже — ближайшие дедлайны и лента договоров с фильтрами по типу/риску/статусу. Те же severity-цвета, что на экране аудита.

## 6. Дедлайны / уведомления

Список ключевых дат, авто-извлечённых из договоров (`KeyData.dates` + классификация: окончание, авто-пролонгация, оплата, уведомление о расторжении). Группировка по срочности (просрочено / 7 дней / 30 дней / позже), на каждой — за сколько дней и куда напоминать (почта / Slack / Telegram). Связь с договором и пунктом-источником в один клик. Это малый шаг над уже существующим `KeyDataExtractor`, но высокая воспринимаемая ценность.

## 7. Доступность и доверие

Severity не кодируется только цветом — всегда есть иконка/подпись (дальтонизм). Контраст текста ≥ 4.5:1. Каждый вывод ИИ обязательно сопровождается `clauseRef` + цитатой — никаких «голословных» рисков. Где ИИ не уверен — явная пометка и приглашение юристу подтвердить/отклонить (human-in-the-loop, заодно собирает метрику точности).
